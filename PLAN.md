# 智能储物柜「刷卡鉴权-开门-回执-UI引导」完整方案（基于当前工程）

## 简要摘要
当前策略：`在线+本地缓存`、`断网禁止开门`、`触摸确认完成`、`一机多门本地选门`、`固定门表`，采用双重链路：
1. `同步鉴权链路`：刷卡后立即请求上级，拿到响应再决定是否开门。  
2. `异步审计链路`：复用现有 `app_uplink` 队列机制上报全过程事件，失败自动重试。  

满足“实时决策开门”，又最大化复用现有 `uplink`/LwIP/LVGL 代码。

## 当前工程进度
1. RC522 驱动已具备寻卡、UID读取、鉴权与块读写能力，但尚未被任务层调用。  
2. 现有 `uplink_poll()` 是异步队列发送模型，不适合“刷卡后必须立刻拿到开门判定”。  
3. 现有响应成功判定逻辑是 `HTTP 2xx && code==0`。  
4. LVGL 已跑通，屏幕为 `800x480`，中文字体可用。  
5. 当前无“门锁执行器 BSP/任务”模块，需要新增。  

## 端到端流程
1. 设备上电后初始化：`RC522`、`门锁BSP`、`Auth任务`、`LVGL页面`、`uplink异步任务`。  
2. 首页显示门位选择（A01~A08，固定门表），用户先点选门位。  
3. 进入“请刷校园卡”状态，`Task_RfidAuth` 轮询读卡。  
4. 读卡成功后提取 `UID`（首版只用 UID，不读扇区），去抖（2秒内同卡同门忽略重复触发）。  
5. 构造同步鉴权请求 `RFID_AUTH_REQ` 并立即 POST 到上级（不入异步队列）。  
6. 上级响应处理规则：  
   - `HTTP 2xx + code==0`：允许开门。  
   - `HTTP 2xx + code!=0`：业务拒绝，不开门。  
   - `HTTP非2xx/超时/解析失败`：网络失败，不开门。  
7. 允许开门时：控制对应门位电磁锁脉冲（默认 1200ms），UI显示“已开门，请取物并关门”。  
8. 用户点击“已取物并关门”后结束会话，返回首页。  
9. 所有关键节点（刷卡开始、鉴权结果、开门结果、会话完成）都作为 `RFID_AUDIT` 异步入队上传，复用现有重试机制。  
10. 本地缓存策略：仅缓存“在线鉴权通过”记录（12小时、RAM LRU 256条），但断网时**仍不放行**，仅用于辅助提示与审计。  

## 上级接口与响应规范
1. 请求路径：首版复用现有 `/api/uplink`。  
2. 同步鉴权请求 `type=RFID_AUTH_REQ`，`payload` 固定字段：
   - `lockerId`（如 `A03`）
   - `uid`（UID十六进制原文）
   - `uidSha1`（UID摘要）
   - `deviceId`
   - `sessionId`
   - `clientTsMs`
3. 响应体至少包含：
   - `code`（int，必须）
   - `msg`（string，可选）
   - `traceId`（string，可选）
4. 业务码约定：
   - `0` 允许开门
   - `1001` 卡未注册
   - `1002` 无该门权限
   - `1003` 当前门位不可取
   - `1004` 重复请求
   - `5001` 服务忙
   - `5002` 系统维护
5. 未识别 `code` 统一按拒绝处理。

## LVGL 显示与操作引导
1. `S_IDLE_SELECT`：门位网格 + 网络状态 + 提示“请选择门位并刷校园卡”。  
2. `S_WAIT_CARD`：高亮已选门位，提示“请将校园卡贴近读卡区”。  
3. `S_READING_CARD`：短时加载（300~800ms），“正在读取校园卡”。  
4. `S_AUTH_PENDING`：加载动画，“正在验证身份，请稍候”。  
5. `S_AUTH_ALLOW_OPENED`：绿色反馈，“验证通过，A03已开门”，主按钮“已取物并关门”。  
6. `S_AUTH_DENY`：红色反馈，按 `code` 显示对应原因，3秒自动返回或“返回重试”。  
7. `S_NET_FAIL`：橙色反馈，“网络异常，暂不可开门”，提供“重试”与“返回”。  
8. `S_SESSION_DONE`：短提示“流程完成”，1秒后回到 `S_IDLE_SELECT`。  

## 需要新增/调整的公共接口与类型
1. 新增 `bsp_locker`（门锁执行器）接口：  
   - `Locker_Init()`  
   - `Locker_Open(locker_index, pulse_ms)`  
2. 新增 `app_auth`（同步鉴权客户端）接口：  
   - `AppAuth_Verify(locker_id, uid, uid_sha1, out_result)`  
3. 新增 `task_rfid_auth`：负责读卡、鉴权、触发开门、派发UI事件。  
4. 扩展 `app_data`：加入会话状态结构（当前门位、最近code、网络状态、会话ID）。  
5. 扩展 `task_lvgl`：由 demo 页面改为状态机页面，接收 `Auth/UI` 事件刷新界面。  
6. 保留现有 `app_uplink`，新增事件类型：`RFID_AUDIT`（异步审计上报）。  

## 测试用例与验收场景
1. 正常放行：选门->刷卡->上级 `code=0`->开门->用户确认->回首页。  
2. 权限拒绝：`code=1002`，不应开门，UI明确拒绝原因。  
3. 网络超时：鉴权超时必须不打开门，UI进入网络异常态。  
4. HTTP 500/非2xx：不打开门，记录审计事件。  
5. 重复刷卡抖动：2秒内同卡同门不重复触发鉴权。  
6. 门锁执行失败：上级放行但硬件开门失败，UI提示故障并上报审计失败。  
7. 审计队列逼近满：业务主链路不阻塞，审计可丢弃并累计丢弃计数。  
8. 多门切换：用户切换门位后刷卡，必须按最新门位鉴权和开门。  
9. 用户不点击“已关门”：45秒会话超时自动结束并回首页。  

## 明确假设与默认值
1. 多门数量默认 `8门`（A01~A08），采用编译期固定映射表。  
2. 首版无门磁传感器，取物完成用“触摸确认完成”。  
3. 首版仅用 UID，不读卡扇区业务数据。  
4. 本地缓存仅 RAM（不落盘），`TTL=12h`，`LRU=256`，仅缓存“放行记录”。  
5. 鉴权超时默认 `send=1500ms, recv=1500ms`。  
6. 门锁开门脉冲默认 `1200ms`。  
7. 现阶段沿用 HTTP，后续可平滑升级 HTTPS。
